The Actual Problem

You are not trying to deploy a billing analytics platform. You are trying to answer: “If retention looks like Company X, pricing looks like Y, and we spend Z on acquisition, what does MRR and burn look like in 12–36 months — and how sensitive is that to churn, viral growth, and win-back?”

That is a scenario engine + curve-fitting workbench, not a Stripe dashboard. Tools like Tidemill (0-star billing-ingest engine) solve a different problem — reconciling historical MRR from live subscriptions. Skip those until you have customers and need actuals reconciliation.

Reframe Open source = libraries and reference implementations you import or study. The deliverable = a small repo of notebooks/scripts where you load benchmark YAML, fit retention, sweep levers, and compare scenarios side-by-side.

Week-1 Plan (No Existing Data)

A realistic path to first predictions without waiting for product telemetry:

DayOutputWhat you're doing
1 Benchmark library (YAML) Encode 5–10 public retention/cost benchmarks (competitors, verticals, case studies) as named scenario presets
2 Retention fit layer Take sparse benchmark points → fitted curve (Theseus or scipy) → monthly retention vector
3 Cohort simulation core Monthly loop: paid acquisition + viral + win-back in, retention decay, MRR out
4 Scenario sweeps Run base / optimistic / pessimistic; tornado chart on churn, K, CAC, ARPU
5 Workbench UI Jupyter for exploration OR Streamlit for sliders; export CSV of all runs
6–7 Fit testing Compare your chosen curve against benchmark points (RMSE); document which competitor each scenario resembles

This is deliberately thin — a data scientist's MVP is a notebook that runs, not a platform. Polish comes after you've argued about assumptions, not before.

Growth Accounting (What the Simulator Computes)

Every scenario engine, whether you write it or wrap libraries, should decompose subscriber change into inspectable flows. This is the minimum physics: 

Active(t) = Active(t-1) + PaidNew(t) + ViralNew(t) + Reactivated(t) - Churned(t)

ViralNew(t) = Active(t-1) × K / viral_cycle_months
Churned(t)  = Active(t-1) × (1 - R(1))   # or apply full cohort retention matrix

MRR(t) = Active(t) × ARPU(t) + Expansion(t) - Contraction(t)

ChurnedPool(t) = ChurnedPool(t-1) + Churned(t) - Reactivated(t)
Reactivated(t) = ChurnedPool(t-1) × winback_rate

Metrics the workbench should output each month

  • Active subscribers, new logos, churned logos, reactivated logos
  • MRR, ARR, net new MRR (waterfall components)
  • Churned pool size
  • CAC spend, viral contribution %, win-back contribution %
  • Gross profit, burn, months of runway (if cash balance provided)

Pricing Structures as Scenario Levers

Without data, pricing enters the model as assumed ARPU and expansion rates, not as inferred behaviour. Encode the pricing model in YAML so you can swap structures:

ModelYAML knobsScenario question it answers
Flat subscription arpu_monthly “What if we're $29/mo like Competitor A?”
Per-seat arpu_monthly, seats_per_account, expansion_rate “Does land-and-expand save us if logo churn is high?”
Freemium free_to_paid_pct, paid_arpu “Can viral K compensate for low conversion?”
Annual prepay annual_mix_pct, annual_discount_pct “Cash upfront vs cliff churn at renewal”
Usage-based arpu_monthly, usage_variance (σ) Sensitivity band on revenue, not point estimate

Working Without Your Own Data

A data scientist without data still works — by anchoring on external benchmarks and making uncertainty explicit. Your week-1 inputs are:

Published retention curves

Eric Seufert / Mobile Dev Memo publishes sparse retention benchmarks (Facebook, etc.) — the same shape Theseus examples use. QuantMar threads and growth essays often cite point-in-time retention at D1/D7/D30.

SaaS benchmark reports

OpenView, KeyBanc SaaS Survey, ChartMogul/Paddle reports: median churn, NRR, CAC payback by ARR band. Use as scenario presets, not gospel.

Competitor S-1 / investor decks

Public companies disclose churn, NRR, ARPU, customer counts at a point in time. Enough to bound a scenario (“at least as good as X”, “worse than Y”).

Synthetic stress tests

Generate cohort matrices with known parameters to verify your simulator matches theory before trusting real benchmarks.

Honest uncertainty With no data, outputs are scenario comparisons, not predictions. The workbench should always show: (1) which benchmark was assumed, (2) fit error on retention points, (3) which levers move the outcome most.

Benchmark Fitting (Competitor & Case Study Numbers)

“Test fit” means: given sparse public retention points for a competitor, how well does your chosen curve reproduce those points, and what 36-month forecast does that imply?

Fit workflow

  1. Encode sparse points in benchmarks/competitor_x.yaml
  2. Fit with one or more methods (Theseus best-fit, scipy Weibull, power law)
  3. Compute RMSE / max error at observed months
  4. Project R(m) for m = 1..36
  5. Run full simulation; tag output fitted_from: competitor_x
  6. Compare multiple competitors side-by-side in a single dataframe

Goodness-of-fit metrics to report

MetricWhat it tells you
RMSE on retention points Overall curve accuracy at benchmark months
Max absolute error Worst month miss — catches bad tail fit
Implied monthly churn @ M12 Sanity check vs published SaaS medians (~0.5–2% monthly for B2B)
Implied LTV = ARPU / churn Compare to competitor-disclosed or inferred unit economics 
# compare_fit.py — conceptual
def fit_quality(observed_months, observed_retention, fitted_fn):
    predicted = [fitted_fn(m) for m in observed_months]
    rmse = (sum((o - p)**2 for o, p in zip(observed_retention, predicted)) / len(observed_months)) ** 0.5
    return {"rmse": rmse, "max_err": max(abs(o - p) for o, p in zip(observed_retention, predicted))}

Theseus Growth — Deep Dive

Use the original repo: github.com/ESeufert/theseus_growth (~214 stars, 29 forks, MIT license, created Jan 2020). The egecakiroglu/theseus_growth fork is a stale copy with 0 stars — install from PyPI (pip install theseus_growth) which tracks the ESeufert package.

Do not confuse with facebookresearch/theseus — that is a PyTorch nonlinear optimization library for robotics, completely unrelated.

What it actually does

  • Takes sparse retention observations (e.g. day 1, 7, 30, 90) as percentages
  • Fits log, exp, linear, quad, Weibull, power, or interpolate — picks lowest error automatically
  • Projects retention to arbitrary horizon
  • Builds forward DAU/MAU from cohort sizes + retention profile
  • Age-segments users by tenure (project_aged_DAU)
  • Exports to Excel/JSON

Learning curve

~2–4 hours to first useful output if you know Python and pandas. ~1 day to wrap it cleanly behind your benchmark YAML loader and produce a 36-month projection. The steepest part is understanding that Theseus projects users and you must own the translation to subscribers/revenue.

Phase 2 — when you have real customers

  • pymc-marketing sBG or cohort Bayesian model for uncertainty bands
  • lifelines on customer-level tenure exports
  • Replace benchmark YAML assumptions with fitted parameters from actuals
  • Consider reading BTYD/CLVTools vignettes if repeat-purchase dynamics appear

What not to do in week 1

  • Deploy Tidemill or any billing-ingest analytics engine
  • Train ML churn classifiers (no labels, no users)
  • Adopt a full FP&A monorepo and retrofit it
  • Let an LLM do the arithmetic (use it to narrate CSV outputs only)
