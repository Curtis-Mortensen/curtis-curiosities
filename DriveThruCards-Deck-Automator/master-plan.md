# Master plan — DriveThruCards Deck Automator

Approved research + implementation plan for automating the partner beta deck builder.

---

## Problem

Publishing a custom US Poker deck on DriveThruCards normally means hand-building a CMYK PDF/X-1a file (back, front, back, front…). DTC now offers a partner beta: **Create US Poker Deck from Image Files Upload**. You upload JPGs; they convert to CMYK and spit out a print PDF. Manual clicking does not scale when you have many cards and multiple back designs, and one deck session tops out around **130 cards**.

## Goal (MVP)

1. Organize card art into **batches by shared back design** (and by the 130-card cap).
2. Drive the beta builder with Playwright: upload backs → edit → upload fronts → edit → assemble → download PDF.
3. Keep each batch ≤ 130 cards; produce one PDF per batch (or one product update per batch).

Out of scope for MVP: full partner-title publishing, tuckbox art, rulebook upload, RGB→CMYK preflight beyond what DTC already does, tests/CI.

## Site findings (2026-07-09)

### Access

- Deep link `…/builder/deck/images/back/6a4ff5a3138d6` redirects to `/builder/deck` when logged out.
- `/builder/deck` renders a Fancybox login popup (`validate_login_credentials.php`). No upload UI without auth.
- `site.drivethrucards.com` is behind Cloudflare; `tools.drivethrucards.com/builder/*` is reachable but session-gated.

### Wizard steps (from UI + `deck.js`)

1. **Upload Back Images** — drag/drop or click; table of previously uploaded images.
2. **Edit Back Images** — warnings, auto-fix (`deck/cleanup/<deck_id>/<image_id>`), reorder.
3. **Upload Front Images** — same upload pattern as backs.
4. **Edit Front Images** — same edit/cleanup pattern.
5. **Assemble My Deck** — check backs + fronts, add cards, set quantities, finish.
6. **Export** — make new product / update product / download only (`next_step`: `make` | `update` | `download`).

### Upload constraints (from the upload notes panel)

| Rule | Value |
|---|---|
| Max file size | 5 MB |
| Formats | JPG / PNG (PNG not recommended — washed-out print) |
| Recommended size | 825 × 1125 px JPEG |
| Color | CMYK preferred; builder converts |
| Safe zone | keep important art ≥ 40 px inside trim |
| Naming hint | `001back.jpg` pairs with `001front.jpg` |

### Print specs (partner help center — final PDF still must satisfy these)

- Page order: back₁, front₁, back₂, front₂, …
- 300 dpi, 1/8" bleed → US Poker layout **2.75" × 3.75"**
- CMYK, ink coverage ≤ 240%
- Export PDF/X-1a:2003, no crop marks

### Deck size

- Builder JS references `deckMaxCards` / `deckMinCards` (values injected per session).
- User target / observed tool cap: **130 cards per deck**.
- Partner banding FAQ allows larger physical decks (up to ~250 before banding splits); treat **130 as the automation hard stop** unless a logged-in session shows a different `deckMaxCards`.

## Card organization by backs

### Why group by back

Assemble pairs selected backs with selected fronts. Shared backs are easiest when:

- one back file is reused for many fronts in the same batch, or
- each card has its own back file but those files are identical art (still upload once if the UI allows reuse — verify when logged in).

Different back designs in one 130-card product are fine if the assemble step can mix them; organization still benefits from **batch folders** so uploads stay small and recoverable.

### Recommended folder scheme

```
cards/
  source/
    backs/
      shared-red.jpg          # one file reused by many cards
      unique/
        012back.jpg
    fronts/
      001front.jpg
      002front.jpg
      …
  manifest.json               # declares cards + which back each uses
  batches/                    # generated by organize-batches.js
    batch-01-shared-red/
      backs/
      fronts/
      batch.json              # list of pairs for assemble
    batch-02-…
```

### Manifest shape

```json
{
  "deckName": "Example Deck",
  "maxCardsPerBatch": 130,
  "cards": [
    { "id": "001", "front": "fronts/001front.jpg", "back": "backs/shared-red.jpg", "qty": 1 },
    { "id": "002", "front": "fronts/002front.jpg", "back": "backs/shared-red.jpg", "qty": 1 },
    { "id": "012", "front": "fronts/012front.jpg", "back": "backs/unique/012back.jpg", "qty": 2 }
  ]
}
```

### Batching rules

1. Sort cards into groups by **back file path** (stable string).
2. Pack groups into batches of ≤ `maxCardsPerBatch` (count = sum of `qty`).
3. Prefer keeping a whole back-group in one batch; split a group only when that group alone exceeds 130.
4. Inside a batch, rename copies to DTC’s suggested sequence: `001back.jpg` / `001front.jpg`, `002…`, so edit/assemble matching is obvious.
5. Write `batch.json` with the pairing table the Playwright assemble step will follow.

### Alternative: one deck, many backs

If you want a single 130-card product with mixed backs, still generate one batch folder, but `batch.json` lists heterogeneous backs. Upload every unique back once, every front once, then assemble by pairing rows. The organizer supports this via `"batchStrategy": "single-deck"` (pack by total qty only, ignore back grouping for split decisions).

## Observed server wait times (user testing)

These are real wall-clock waits on DTC’s side, not Playwright overhead:

| Job | Approx. duration |
|---|---|
| Auto-correct / fix colors | ~10 minutes |
| Render cards onto a sheet | ~10 minutes |
| Publish (vs download-only) | ~10 minutes |

Automation must tolerate **≥20 minutes per long job** and detect completion via page shifts (see below), not fixed `sleep(600000)`.

## Playwright architecture (lean)

One file per action / concept:

| File | Job |
|---|---|
| `src/config.js` | URLs, timeouts, paths, max cards, Edge channel |
| `scripts/auth-save.js` | headed Edge login → persistent `auth/edge-profile/` |
| `src/browser.js` | launch installed Edge (`channel: 'msedge'`), headed, watchable |
| `src/waits.js` | page-shift waits + 30s heartbeats for long DTC jobs |
| `src/organize-batches.js` | manifest → `cards/batches/*` |
| `src/steps/upload-images.js` | drag/drop or file chooser on upload step |
| `src/steps/edit-images.js` | wait for auto-correct; click Next |
| `src/steps/assemble-deck.js` | pair backs/fronts from `batch.json`, set qty |
| `src/steps/export-pdf.js` | download PDF **or** publish (`make` / `update`); long waits |
| `src/run-batch.js` | glue: open Edge, walk one batch through the wizard |

### Browser: Microsoft Edge (headed)

- **Not headless.** Default `headless: false` so you can watch.
- Uses Playwright `chromium.launchPersistentContext` with `channel: 'msedge'` (your installed Edge).
- Profile directory: `auth/edge-profile/` (gitignored). Login once via `npm run auth:save`; later runs reuse cookies.
- Optional `--keep-open` leaves the window open after a run for inspection.

### Auth approach

1. `npm run auth:save` opens Edge with the project profile; human logs into partner tools.
2. Profile folder on disk is the source of truth (optional cookie snapshot also written).
3. Re-run auth save if the session expires.

Do not put email/password in the repo.

### How waits work (page shifts, not blind sleeps)

Playwright continues when the **page state changes**, for example:

1. **URL change** — `page.waitForURL(...)` when the wizard advances.
2. **Selector appears** — success UI, Next button, assemble checkboxes, etc.
3. **Busy UI clears** — spinners / `.blockUI` / “please wait” / `.batch-wait` go hidden.
4. **Download event** — for `--mode download`, wait for Edge to start a file download.

`src/waits.js` wraps these with a **20-minute** ceiling (`longJobTimeoutMs`) and a **30-second heartbeat** log line so a 10-minute color-correct job does not look frozen. Edit + export steps call `waitForPageShift` after clicks that start server work.

### Selector strategy (to harden after login)

From public CSS/JS we already know useful hooks:

- Progress: `.progress-status .step` / `.active`
- Nav: `.submit-next`, red Back/Next buttons
- Upload page body class: `.page-deck-images`
- Uploaded table: `.deck-uploaded`, `.uploaded-images`
- Edit/lists: `.page-deck-lists`, `.image-fix`, `.batch-edit`, `.submit-next`
- Assemble: `.page-deck-assemble`, `.checkboxes-backs`, `.checkboxes-fronts`, `.submit-add`, `.card-quantity`, `.submit-wrapper-finish`
- Export: `.page-deck-export`, `input[name="next_step"]` values `make` | `update` | `download`

Upload dropzone text observed: “Drag and drop files here or click to upload.” Prefer `page.getByText(...)` + `locator('input[type=file]')` once visible after login.

### Run loop (one batch)

```
Edge profile → open start URL (live …/images/back/<id> preferred)
→ upload all batch backs
→ Next → wait page shift to edit → auto-correct if offered (≤20 min) → Next
→ wait page shift to front upload → upload fronts
→ Next → edit fronts (same long wait pattern)
→ wait page shift to assemble → pair from batch.json → Finish
→ export: download PDF and/or publish (each may take ~10 min)
→ write run-log.json
```

## Implementation phases

1. **Research + skeleton** (this folder) — done.
2. **Edge + long-wait helpers + HTML explainer** — done.
3. **Logged-in selector pass** — with real credentials in Edge, dump DOM for upload/assemble pages; lock selectors.
4. **Organize + dry upload** — one small batch (2–3 cards), watch Edge through auto-correct.
5. **Assemble + PDF download / publish** — confirm export path; tune busy selectors if needed.
6. **Multi-batch runner** — loop batches; stop on failure; resume from last good step.

## Risks / assumptions

- Beta UI may change; keep selectors in one place.
- Cloudflare on `site.*` may interfere with login from some networks; prefer logging in on `tools.drivethrucards.com`.
- Unverified partner accounts may lock some tools (seen in partner dashboard screenshot).
- Long jobs (~10 min) are expected; busy-selector list may need tightening after a live run.
- Edge must be installed locally (`msedge` channel). On machines without Edge, switch `browserChannel` only if the user asks.
- Assumption: 130 is the builder max unless `deckMaxCards` says otherwise in-session.
- Assumption: one uploaded back image can be checked once and paired with many fronts during assemble (verify; if not, duplicate back files per card in the batch folder).

## Success criteria

- From a filled `manifest.json`, `organize-batches.js` produces valid batch folders ≤ 130 cards each.
- With an Edge profile login, `run-batch.js` can take one batch from empty upload through PDF download or publish without manual clicking (aside from initial auth save), while the human can watch the window.
- Long jobs print heartbeats and proceed only after a detected page shift (or download).
- `docs/plan-explainer.html` opens offline and explains the plan in plain language.
- `dev-log.md` records every selector discovery and failure.
