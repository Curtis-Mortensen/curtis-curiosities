1. Introduction and Aims

Research-Bot agents produce long, self-contained HTML research reports with embedded CSS. Those files are easy to open in a browser, but sharing, archiving, printing, and attaching them often wants a PDF. This document answers one question:

What are the reliable options for converting Research-Bot HTML reports into PDF, and which approach best fits this corpus?
Central thesis: For these reports — static, self-contained HTML with modern CSS (custom properties, grid, tables, occasional SVG), almost no JavaScript, and mostly dark on-screen themes — a Chromium print pipeline is the most reliable default. That covers ordinary browser “Print → Save as PDF,” Chrome/Chromium’s --print-to-pdf CLI, Playwright/Puppeteer, and self-hosted services such as Gotenberg. Custom CSS engines (WeasyPrint, Prince, Vivliostyle) can produce nicer pagination, but they are a different rendering engine than the browser the reports were designed in. Archived tools built on old WebKit or PhantomJS (especially wkhtmltopdf and the npm html-pdf package) should not be used for new work.[1][2][3]

Aims of this brief:

  • Map the main technical families of HTML→PDF conversion and what each is good at.
  • Ground recommendations in how Research-Bot HTML files are actually built.
  • Give a concrete catalog of tools (commercial and open source) with trade-offs.
  • Recommend a lean “reliable enough” path for batch conversion without building a product.

2. Limitations and Evidence Strength

Theoretical limitations first. “Look the same as the browser” and “look like a printed book” are different goals. Web layout is continuous and interactive; PDF is discrete pages. No converter can perfectly satisfy both without trade-offs around page breaks, headers/footers, dark themes, and ink-friendly print CSS. Human taste (“good enough PDF”) is partly subjective.[4]

Method / evidence limitations. This report synthesizes vendor docs, comparison articles, GitHub/project status pages, W3C/Chrome documentation, and one local Chrome headless probe of a Research-Bot file. It is not a controlled multi-engine visual regression suite across all 29 reports. Vendor comparison posts (Apryse, Iron Software, PDF4.dev, DocRaptor) are useful for catalogs and dates but are commercially interested; claims about “fastest” or “best” were cross-checked against primary project pages where possible.[5][2]

Field maturity. HTML-to-PDF is a mature, crowded niche with recurring tool death (PhantomJS, wkhtmltopdf) and a durable architectural split between “run a real browser” and “implement CSS for print.” CSS Paged Media support in browsers improved sharply around Chrome 131 / Safari 18.2 (late 2024), so older advice that “browsers cannot do margin boxes” is partly outdated for Chromium.[6][7]

3. Precursor and Subset Questions

Before picking a tool, smaller questions decide the answer:

1. Fidelity vs. pagination

Do you want the PDF to match Chrome’s on-screen/print layout, or book-style footnotes, running headers, and strict page models? Chromium wins the first; Prince/WeasyPrint/Vivliostyle lean toward the second.[1][4]

2. One-off vs. batch

Manual Print-to-PDF is enough for occasional exports. Batch conversion of dozens of reports wants a CLI or small script (Chrome headless, Playwright, or Gotenberg).[3]

3. JavaScript required?

If charts or frameworks render only after JS, you need a browser engine. Research-Bot reports surveyed here have zero <script> tags, so JS execution is optional — but Chromium still helps for CSS fidelity.

4. Dark theme → paper

Most reports use a dark GitHub-like theme. Without @media print rules (or printBackground choices), PDFs can be ink-heavy black pages or lose intended colors. Only a minority of reports already ship print CSS.

5. Who operates the renderer?

Self-host Chromium (CLI / Playwright / Gotenberg) vs. pay a managed API (DocRaptor, Browserless, PDFShift, etc.). For a personal research folder, self-host is usually enough.[1][8]

6. Security surface

Archived engines with unpatched CVEs are unacceptable if HTML is ever untrusted. Even for trusted local files, dead tooling is a maintenance trap.[1][9]

5. History of HTML-to-PDF as a Field

5.1 From print stylesheets to dedicated engines

Browsers have long supported “print this page,” and CSS evolved print-oriented modules (Paged Media, Generated Content for Paged Media). In practice, browser support for advanced pagination lagged for years, so specialized engines filled the gap.[6]

PrinceXML (YesLogic, first release 2003) became the reference commercial CSS-to-PDF engine; CSS co-creator Håkon Wium Lie has been associated with the project. DocRaptor later productized Prince as a hosted API.[13][14] Other commercial print engines include PDFreactor and Antenna House Formatter.[2]

5.2 The “embed a browser” era

On the open-source side, wkhtmltopdf (wrapping Qt WebKit) became the default CLI for a decade of Rails/PHP/Node apps. Parallel paths included PhantomJS-based Node packages such as html-pdf. Both underlying engines were eventually abandoned: PhantomJS (announced 2018), wkhtmltopdf (GitHub archive early 2023; last stable 0.12.6 in 2020), with critical unpatched issues such as CVE-2022-35583 (SSRF, CVSS 9.8).[1][15][9]

Headless Chrome, then Puppeteer (Google) and Playwright (Microsoft), became the modern replacements: drive a current Chromium and call the same print-to-PDF path the browser uses.[16][3] Gotenberg packaged Chromium (plus LibreOffice) as a Docker HTTP microservice for language-agnostic conversion.[8]

5.3 Lightweight CSS engines and publishing tools

WeasyPrint (Python, Cairo/Pango) implements its own layout engine aimed at print CSS without shipping a browser. Pandoc has moved HTML→PDF defaults away from wkhtmltopdf toward WeasyPrint for installability.[17][18] Open publishing stacks such as Vivliostyle and the Paged.js polyfill target book/magazine pagination on top of web tech.[19][7]

5.4 Recent browser catch-up

Chrome 131 (November 2024) added CSS page margin boxes for print; Safari 18.2 followed. That narrows — but does not eliminate — the gap with Prince-class engines for headers, footers, and page counters. Firefox still lagged on margin boxes as of early 2026 commentary.[6][20]

6. Five Families of Conversion Approaches

A. Interactive browser print

Open the HTML, Print, “Save as PDF.” Zero setup. Quality depends on the browser’s print dialog and your @media print CSS. Best for one-offs.[6]

B. Headless Chromium automation

Same engine as Chrome, scripted: CLI --print-to-pdf, Puppeteer, Playwright, Selenium+CDP, Gotenberg. Highest fidelity to how the report looks in Chrome.[3][16]

C. Custom CSS print engines

WeasyPrint, Prince, PDFreactor, Antenna House, Vivliostyle. Stronger pagination features; layout may diverge from Blink/WebKit. License cost varies from free to thousands/year.[2]

D. Managed APIs

DocRaptor (Prince), Browserless, PDFShift, PDFCrowd, PDF4.dev, etc. You POST HTML/URL; they return PDF. Good when you refuse to run Chromium yourself.[13][21]

E. Non-HTML rebuilders

Pandoc→LaTeX, pdfmake, PDFKit, jsPDF/html2pdf.js. Either rewrite content into another format or screenshot the DOM. Usually a poor fit for preserving existing Research-Bot HTML as-is.[2]

8.1 Practical Chromium knobs that matter

  • printBackground / background graphics: needed if you want dark theme cards and colored table headers in the PDF; for paper printing you usually want print CSS that forces light backgrounds instead.[16]
  • preferCSSPageSize / @page: lets CSS control paper size when present.
  • --no-pdf-header-footer: strips Chrome’s default date/URL chrome from CLI PDFs.[3]
  • Margins & page numbers: either CSS margin boxes (Chromium 131+) or Playwright/Puppeteer header/footer HTML templates.[6]

9. Local Probe on a Research-Bot Report

To sanity-check the Chromium path against a real report (not a vendor demo), this research run converted management-theory-research.html (~84 KB, includes @media print) with Google Chrome headless on Linux:

google-chrome-stable --headless=new --disable-gpu --no-sandbox \
  --no-pdf-header-footer \
  --print-to-pdf=management-theory.pdf \
  "file:///…/Research-Bot/management-theory-research.html"

10. Recommendation for This Folder

Reliable default: use a Chromium print pipeline, and invest a little in shared @media print CSS so PDFs are readable on paper.

  1. Day-to-day / one file: open in Chrome → Print → Save as PDF (disable browser headers/footers; enable or disable background graphics intentionally).
  2. Batch / repeatable: Chrome --print-to-pdf in a small shell loop, or a short Playwright script if you want margins, printBackground, and waiting hooks.
  3. If you later want a service: Gotenberg in Docker — same Chromium fidelity, HTTP for any language.[8]
  4. Skip for now: wkhtmltopdf, Phantom-based packages, and paid Prince/DocRaptor unless you decide these reports should become publication-layout books.
  5. Optional experiment: WeasyPrint on one report to compare pagination — keep Chromium if layout diverges on grid/cards.

10.1 Suggested print-CSS baseline for future reports

Most reports still lack print rules. A minimal pattern already used in a few files:

@media print {
  body { background: white; color: black; }
  .overview-card, .thesis-box, table { break-inside: avoid; }
  a { color: #0645ad; }
}

Stronger future variants can set @page { margin: 18mm; }, lighten borders, and force link URLs to appear after citations if you care about print-only bibliography UX. Chrome’s newer margin boxes can add page numbers without Playwright templates once you standardize on Chromium.[6]

10.2 Example batch loop (Chrome CLI)

# Convert every report in the Research-Bot tree into ./pdf-out/
mkdir -p pdf-out
find . -name '*.html' -print0 | while IFS= read -r -d '' f; do
  out="pdf-out/${f#./}"
  mkdir -p "$(dirname "$out")"
  google-chrome-stable --headless=new --disable-gpu --no-sandbox \
    --no-pdf-header-footer \
    --print-to-pdf="$PWD/${out%.html}.pdf" \
    "file://$PWD/$f"
done

Note: flags like --no-sandbox are environment-specific (containers/CI). On a normal desktop install they are often unnecessary. Prefer absolute file:// URLs and absolute PDF output paths.[3]

11. Conclusion and Further Research

HTML→PDF is no longer a “pick wkhtmltopdf” problem. The reliable path for modern, self-contained research HTML is the same engine that already displays the reports: Chromium’s print stack — manually, via CLI, via Playwright/Puppeteer, or via Gotenberg. Custom engines still win when you are typesetting a book; managed APIs win when you refuse to operate a browser. Archived WebKit/Phantom tooling is a dead end.

For Research-Bot specifically, reliability is less about finding exotic software and more about (1) standardizing print CSS across reports and (2) picking one Chromium automation path for batch exports.

Further research questions

  • Should Research-Bot adopt a shared print stylesheet snippet as part of the agent instructions in README.md?
  • Do authors want PDFs to preserve the dark theme (screen archive) or force a light theme (paper/share)?
  • Is clickable link preservation and PDF outline/bookmarks (from h1/h2) worth requiring Playwright over raw CLI?
  • How do SVG diagrams and wide tables fragment across page breaks in Chromium vs WeasyPrint on the Apocalypse-Story set?
  • If reports are ever published publicly, do PDF/A or PDF/UA requirements appear (favoring Prince/WeasyPrint/commercial toolkits)?
  • Should PDFs be committed next to HTML, gitignored, or generated only on demand?
